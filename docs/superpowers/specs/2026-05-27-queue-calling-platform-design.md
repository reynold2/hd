# 等锅叫号平台完整设计

## 目标

建设一个适用于麻辣烫、串串、冒菜、炸串等现场自选等餐场景的扫码排号叫号平台，包含顾客端 uni-app 小程序、商家 Web 后台、平台管理后台、店员/制作/收银工作台和大屏叫号页。

## 核心业务理解

系统核心不是传统“先点单付款”，而是“号码 = 一次现场消费篮/餐次”。

顾客到店后先自选菜，接待或店员给一个号码，并可录入初始菜篮信息。顾客在小程序扫码或输入号码绑定本次餐次后，可以查看等待进度、补充备注、追加饮料或配料、呼叫服务、参与等待互动。菜篮被贴号送到制作端，厨师按号码制作并叫号。顾客取餐或由店员送达后，号码仍保持占用。顾客吃完呼叫结账，收银员汇总初始菜篮和顾客主动追加项生成总额，展示门店收款二维码。顾客线下付款后，收银员点击“已收到”，本次号码结束并进入统计。

## 主流程

1. 顾客到店自选菜。
2. 接待或店员发号，录入人数、口味、备注、初始菜品或金额参考。
3. 顾客扫码或输入号码绑定餐次。
4. 顾客等待期间查看前方人数、预计等待时间、制作进度和当前叫号。
5. 顾客可追加饮料、配料、备注或呼叫服务，追加项归属当前号码。
6. 制作端看到号码、备注、菜篮内容，执行开始制作、制作完成、叫号。
7. 顾客端收到叫号提醒，去取餐或等待店员送达。
8. 顾客取餐后进入用餐中，未结账前号码继续占用。
9. 顾客呼叫结账，收银端汇总初始菜篮和追加项。
10. 收银员展示门店支付二维码，顾客付款后收银员确认已收款。
11. 号码进入已完成，释放占用，生成统计和操作日志。

## 号码状态

- `idle`：空闲号码。
- `occupied`：已占号，待制作。
- `preparing`：制作中。
- `called`：已叫号，待取餐。
- `dining`：已取餐或已送达，用餐中。
- `checkout_requested`：顾客已呼叫结账。
- `paid`：收银员确认已收款。
- `completed`：餐次完成，号码释放。
- `skipped`：暂时跳过。
- `cancelled`：取消或作废。
- `risk_unpaid`：疑似未付款离店风险。

## 顾客端

技术：uni-app + Vue。

页面包括门店首页、取号/绑定号码、取号成功、排队进度、当前叫号、号码详情、补充备注、呼叫服务、菜品展示、等待互动、我的号码、门店信息。

顾客端不直接完成在线支付。支付动作发生在线下扫码，系统只接收收银员“已收到”的确认结果。

## 商家端

技术：Vue 3 + Vite + Element Plus。

模块包括首页概览、队列管理、号码详情、手动发号、扫码取号、制作工作台、收银记账、菜品管理、门店管理、员工权限、排号规则、二维码生成、大屏叫号、数据统计、操作日志、风控限制、系统设置。

收银记账只记录应收金额、追加项、优惠、支付方式和确认收款状态，不直连微信或支付宝官方支付，不做交接班。

## 平台管理端

平台管理员模块包括商家入驻审核、门店审核、账号管理、角色权限、套餐权限、系统公告、投诉反馈、平台配置、平台统计和全局操作日志。

## UI 风格

参考原型图执行：

- 顾客端使用橙红主色，大号号码卡片，进度时间线，信息清晰。
- 店员/制作端使用绿色操作态，大按钮，突出当前号码和备注。
- 商家后台使用深色侧边栏、白色内容卡片、高密度表格。
- 平台后台偏蓝色 SaaS 风格。
- 大屏叫号使用深色高对比，适合远距离查看。

## 技术架构

- 后端：Python 3.8 + FastAPI + SQLAlchemy + MySQL + Redis。
- 管理端：Vue 3 + Vite + Element Plus。
- 顾客端：uni-app + Vue。
- 实时更新：首版使用短轮询，叫号和大屏接口预留 WebSocket。
- 权限：平台、商家、门店、员工多层角色权限。
- 数据隔离：所有商家业务数据带 `merchant_id`，门店业务数据带 `store_id`。

## 首版交付范围

首版一次性覆盖完整多商户平台，但实现深度以主流程可用为优先。支付、退款、官方对账、交接班、库存扣减不进入首版。
