# 角色登录、员工注册与后台权限体系设计

## 背景

当前项目已经具备顾客端首页、门店目录、流程验证面板，以及后台管理界面的基础路由和登录入口。但随着业务继续扩展，权限体系需要进一步统一，新增以下能力：

1. 登录只保留账号和密码，不做注册入口，但要具备必要的安全校验。
2. 系统管理员和平台管理员可以在后台管理界面维护用户账号、密码、头像，并可将账号关联到对应微信账户。
3. 老板端小程序个人中心增加“添加员工二维码”，员工扫码后可与门店建立关联，在扫码页内设置账号、密码、头像，选择所属门店内角色；若账号不存在则可创建，若账号已存在则提示并进入绑定/关联流程。
4. 老板账号只能管理自己门店的员工；系统管理员和平台管理员可以跨门店修改这家店的员工信息。
5. 员工完成扫码注册或绑定后，应自动进入对应权限首页。

本设计采用“统一身份中心 + 门店隔离权限 + 扫码绑定流程 + 后台账号管理”的方案，保证小程序与后台共享一套可维护的账号和权限模型。

## 目标

1. 统一用户、角色、门店、微信账户的关联模型。
2. 后台登录仅保留账号密码入口，不提供用户自助注册。
3. 系统管理员、平台管理员可维护账号、密码、头像及微信绑定。
4. 老板可管理自己门店员工的账号、密码、头像、删除与角色调整。
5. 员工扫码后能完成门店关联、角色选择、账号创建或绑定，并自动进入对应权限首页。
6. 保持真实接口驱动，小程序与后台都从后端获取数据。

## 范围

### 本期包含

- 后台登录页（账号密码登录）
- 登录安全校验与角色路由
- 后台用户管理页
- 账号、密码、头像、微信账户关联能力
- 门店员工管理权限隔离
- 老板端小程序个人中心的员工二维码入口
- 员工扫码注册 / 绑定流程
- 扫码后角色选择与自动跳转
- 种子数据与可维护初始化脚本

### 本期不包含

- 第三方统一登录（SSO）
- 多因子认证
- 复杂组织架构（集团、品牌矩阵）
- 复杂权限编排器
- 实名认证、风控、审计后台

## 总体方案

采用统一身份中心作为单一事实来源：

- 统一管理用户、角色、门店、微信账户绑定关系
- 登录入口分为后台登录和小程序扫码入口两类
- 后台登录仅允许账号密码，不做注册
- 小程序员工扫码后允许进行门店关联与账号创建/绑定
- 登录成功后由后端返回角色、门店、首页路由与可见页面

这样可以保证：

- 小程序和后台都能复用同一套账号数据
- 老板只能管理自己门店的员工
- 系统管理员 / 平台管理员可跨门店维护员工信息
- 员工扫码后能够自然进入对应角色页面

## 角色体系设计

### 小程序端角色

- 顾客
- 老板
- 服务员
- 制作
- 收银
- 员工扫码注册态（临时态，仅在扫码页出现）

### 后台管理端角色

- 平台管理员
- 门店老板
- 收银
- 大屏

### 角色说明

- **顾客**：查看门店、选菜、绑定分配号、等待、加退菜、呼叫店员、支付
- **老板**：查看门店经营数据、订单状态、门店设置、员工管理
- **服务员**：处理顾客呼叫、协助加退菜、备注和沟通
- **制作**：查看制作队列、制作状态、催菜提醒
- **收银**：处理结算、核对金额、出示二维码、确认支付
- **平台管理员**：管理平台级账号、门店、角色、数据初始化和跨门店员工维护
- **门店老板**：管理本门店数据与员工账号
- **大屏**：展示门店实时状态和队列信息

## 身份模型

建议后端至少维护以下实体。

### User（统一用户）

字段建议：

- `id`
- `username`
- `display_name`
- `avatar_url`
- `user_type`：测试 / 生产
- `status`：启用 / 停用 / 待绑定
- `created_at`
- `updated_at`

### Credential（登录凭证）

字段建议：

- `user_id`
- `login_name`
- `password_hash`
- `provider`：后台 / 小程序 / 员工扫码
- `openid`：小程序绑定时使用
- `last_login_at`
- `password_updated_at`

### RoleAssignment（角色分配）

字段建议：

- `user_id`
- `role_code`
- `scope_type`：平台 / 门店
- `store_id`
- `enabled`

### StoreBinding（门店绑定）

字段建议：

- `user_id`
- `store_id`
- `relationship_type`：老板 / 收银 / 大屏 / 服务员 / 制作 / 顾客
- `source`：扫码注册 / 管理员创建 / 手工配置

### WechatBinding（微信绑定）

字段建议：

- `user_id`
- `openid`
- `unionid`
- `source`
- `enabled`

## 登录与安全校验

### 后台登录

只保留账号和密码登录，不提供注册入口。

必要的安全校验包括：

- 密码不能为空
- 账号不能为空
- 密码最小长度校验
- 登录失败次数限制或短时重试限制
- 输入内容基础清理（防空格、非法字符）
- 登录后返回后端签发的会话标识或令牌

### 小程序扫码入口

扫码入口仅用于联调和业务引导，不承担真实数据存储。

必要的安全校验包括：

- 校验门店二维码有效性
- 校验门店状态是否可用
- 校验当前 openid 是否已绑定或待绑定
- 校验扫码注册权限是否允许

## 路由策略

### 小程序端

#### 正式模式

- 小程序启动后识别当前 `openid`
- 后端查询该 `openid` 对应的身份与门店角色
- 如果存在有效角色：直接进入对应角色页面
- 如果不存在绑定：进入授权或绑定页
- 真实模式下不显示测试入口

#### 扫码注册模式

- 用户扫描老板端生成的员工二维码
- 扫码后进入员工注册 / 绑定页
- 选择所属门店内角色
- 输入或确认账号、密码、头像
- 若账号不存在，则创建并绑定
- 若账号已存在，则根据权限进行绑定或提示重复
- 成功后自动进入对应权限首页

### 后台管理端

- 统一登录页输入账号密码
- 后端验证通过后返回角色和门店上下文
- 根据角色自动路由：
  - 平台管理员 → 平台后台首页
  - 门店老板 → 门店运营首页
  - 收银 → 收银工作台
  - 大屏 → 大屏展示页

## 后台登录页设计

### 目标

为后台管理提供统一入口，避免未登录时直接暴露管理页面。

### 展示内容

- 账号输入框
- 密码输入框
- 登录按钮
- 环境提示
- 角色说明

### 登录后路由

- 平台管理员：平台级总览、门店管理、用户管理、数据初始化
- 门店老板：本店经营、订单管理、员工管理
- 收银：收银台、支付核对、待结账列表
- 大屏：实时看板和队列信息

## 用户管理设计

### 管理入口

系统管理员、平台管理员、门店老板都可进入用户管理，但权限不同：

- **系统管理员 / 平台管理员**：可管理所有门店用户信息
- **门店老板**：只能管理自己门店的员工

### 可执行操作

- 创建员工账号
- 修改账号
- 修改密码
- 修改头像
- 绑定或解绑微信账户
- 选择所属门店内角色
- 停用 / 启用账号
- 删除自己门店员工（门店老板仅限本店）

### 表单规则

- 账号可默认生成，允许手动修改
- 密码可默认生成，若不修改则使用默认值
- 头像可默认使用系统头像，允许替换
- 保存前做重复账号校验
- 保存后返回新的角色路由

## 老板端小程序个人中心设计

### 目标

让老板在小程序端快速生成员工二维码，完成员工与门店的关联。

### 功能区块

- 我的门店
- 员工管理入口
- 添加员工二维码
- 员工列表
- 员工角色概览

### 添加员工二维码流程

1. 老板点击“添加员工二维码”
2. 系统生成带门店与邀请信息的二维码
3. 员工扫码进入注册 / 绑定页
4. 员工选择门店内角色
5. 员工填写账号、密码、头像
6. 若账号不存在则创建，存在则提示已注册或允许绑定
7. 完成后自动进入对应权限首页

### 二维码内容建议

- 门店 ID
- 邀请令牌
- 过期时间
- 发起人用户 ID
- 可选角色范围

## 员工扫码注册 / 绑定页设计

### 展示内容

- 门店名称
- 可选角色列表（门店内）
- 账号输入
- 密码输入
- 头像上传或选择
- 确认按钮

### 交互规则

- 默认值可直接提交
- 只有当用户要修改时才需要改动
- 账号不存在时，点击确认创建账号并绑定门店
- 账号已存在时，提示已注册，允许按权限进行绑定或提示联系管理员
- 成功后自动跳转到对应角色首页

### 结果提示

- 注册成功
- 绑定成功
- 账号已存在
- 权限不足
- 门店二维码失效

## 权限与可见性规则

- 只有平台管理员可见平台级管理功能
- 门店老板只能看见自己门店的数据与员工
- 系统管理员可跨门店维护员工
- 收银只能看见收银相关页面和操作
- 大屏只能看见展示型页面
- 小程序端按角色限制菜单与动作
- 扫码注册页仅对受邀请用户或合法扫码来源开放

## 测试数据设计

### 种子数据原则

1. 默认可一键初始化
2. 每个关键角色至少一个有效账号
3. 每个测试账号都能关联真实门店数据
4. 数据命名清晰可识别
5. 测试数据可重复执行且幂等

### 建议种子数据

#### 平台级

- 平台管理员 1 个

#### 门店级

每个门店至少创建：

- 门店老板 1 个
- 收银 1 个
- 大屏 1 个
- 服务员 1 个
- 制作 1 个
- 顾客 1 个测试身份
- 员工扫码邀请二维码配置 1 套

### 初始密码策略

- 默认密码仅用于初始化
- 保存时若未修改可沿用默认值
- 生产环境账号不得直接暴露默认密码

## 后端接口建议

### 登录相关

- `POST /api/admin/login`：后台登录
- `GET /api/admin/me`：后台当前身份与角色
- `GET /api/admin/home`：后台首页配置

### 账号与员工管理

- `GET /api/admin/users`：用户列表
- `POST /api/admin/users`：创建用户
- `PATCH /api/admin/users/:id`：修改用户信息
- `DELETE /api/admin/users/:id`：删除用户
- `PATCH /api/admin/users/:id/password`：修改密码
- `PATCH /api/admin/users/:id/avatar`：修改头像
- `PATCH /api/admin/users/:id/wechat-binding`：绑定微信账户

### 门店员工邀请

- `POST /api/stores/:id/employee-invitations`：生成员工二维码
- `GET /api/employee-invitations/:token`：扫码获取邀请详情
- `POST /api/employee-invitations/:token/accept`：员工确认绑定并注册

### 小程序相关

- `POST /api/auth/wechat-login`：微信登录并绑定 openid
- `GET /api/me`：获取当前身份和角色
- `GET /api/me/home`：获取当前角色首页配置
- `GET /api/roles/resolve?openid=...`：根据 openid 解析角色

### 数据初始化

- `POST /api/admin/seed-test-data`
- `POST /api/admin/reset-test-data`

## 验收标准

1. 后台登录只保留账号密码入口，并具备基础安全校验。
2. 系统管理员和平台管理员可以在后台管理界面修改账号、密码、头像和微信关联。
3. 门店老板只能管理自己门店员工，不能修改其他门店员工。
4. 老板端小程序个人中心可生成员工二维码。
5. 员工扫码后可选择所属门店内角色，并填写账号、密码、头像。
6. 账号不存在时可创建，存在时能提示已存在并按规则处理。
7. 完成注册或绑定后，员工会自动进入对应权限首页。
8. 后端种子数据可一键初始化，并可被系统管理员维护。
9. 小程序和后台都从真实接口获取数据，不依赖本地假数据。

## 实现约束

- 不能把生产逻辑和测试逻辑混在同一个本地假数据层中，必须由后端接口决定。
- 用户和门店的关系必须有明确的权限边界，门店老板不得越权。
- 扫码注册流程必须支持幂等处理，避免重复创建账号。
- 角色路由必须可扩展，后续增加新角色时不破坏现有结构。

## 推荐方案

采用“统一身份中心 + 门店隔离权限 + 扫码绑定流程 + 后台账号管理”的方案。

原因：

- 既能支持真实业务登录，又能支持员工扫码注册
- 门店老板和系统管理员的权限边界清晰
- 小程序、后台、扫码入口可复用同一套数据模型
- 后续扩展员工、门店和角色不会增加大量重复逻辑

## 后续实施拆分建议

建议按以下顺序实施：

1. 先完成统一身份与门店权限模型
2. 再实现后台用户管理与登录安全校验
3. 然后实现老板端员工二维码入口与扫码注册页
4. 接着把小程序和后台的真实接口全部接通
5. 最后补齐种子数据初始化与权限边界测试
