# 顾客端流程检验面板设计

## 背景

当前 `customer-miniapp/src/pages/index/index.vue` 只展示了门店概览、号码绑定入口和少量占位信息，尚不足以覆盖完整业务链路的状态验证。用户希望增加一套“检验流程”，用于按业务顺序验证顾客端与店员端之间的关键状态流转，覆盖：

- 顾客进店
- 选菜
- 扫码 / 输入号码 / 直接绑定分配号
- 制作中
- 等待期间小游戏互动
- 中途加菜 / 退菜
- 呼叫店员
- 店员端加菜 / 退菜
- 呼叫店员算账
- 店员确认金额
- 提供二维码
- 店员确认支付

本设计采用“单页步骤面板 + 事件模拟器”的方式实现状态检验，优先满足快速验收与后续扩展。

## 目标

1. 验证完整主流程是否能够按顺序流转。
2. 验证每个关键动作在当前状态下是否允许执行。
3. 验证加菜、退菜、算账、支付等关键节点的前置条件和结果回写。
4. 为后续扩展异常分支、多人桌台、优惠券、催菜等场景留出结构。

## 范围

### 本期包含

- 单页流程状态面板
- 当前状态详情区
- 事件模拟按钮区
- 状态日志区
- 基础异常提示
- 场景扩展占位入口

### 本期不包含

- 真实后端联调
- 服务端持久化
- 真正的支付接口接入
- 复杂权限体系
- 多页面重构

## 核心交互结构

### 1. 顶部信息区

展示：

- 门店名称
- 当前订单 / 分配号
- 当前流程状态
- 进度摘要

用途：让测试者一眼看到当前处于哪一步。

### 2. 左侧主流程步骤条

按业务顺序展示状态节点：

1. 进店
2. 选菜
3. 扫码 / 输入 / 绑定分配号
4. 制作中
5. 等待小游戏互动
6. 中途加菜
7. 中途退菜
8. 呼叫店员
9. 店员端加菜 / 退菜
10. 呼叫店员算账
11. 店员确认金额
12. 提供二维码
13. 店员确认支付
14. 完成

每个节点支持：

- 点击切换到该状态（仅用于模拟）
- 显示已完成 / 当前 / 未达成状态
- 展示关键依赖条件

### 3. 右侧状态详情区

用于显示当前状态的验证信息：

- 当前状态说明
- 前置条件
- 允许动作
- 禁止动作
- 最近一次校验结果
- 当前金额 / 菜品数量 / 绑定号码

这个区域是“检验”的核心，重点回答：

- 为什么能进入这一步？
- 为什么不能执行某动作？
- 动作执行后系统状态有没有变化？

### 4. 底部事件模拟器

提供按钮或快捷操作，模拟真实事件：

- 模拟进店
- 选菜
- 扫码绑定
- 输入分配号绑定
- 直接绑定分配号
- 进入制作中
- 开始小游戏互动
- 加菜
- 退菜
- 呼叫店员
- 店员确认加菜
- 店员确认退菜
- 呼叫店员算账
- 店员确认金额
- 生成二维码
- 支付成功
- 支付失败重试

事件触发后：

- 先校验当前状态是否合法
- 合法则推进状态并记录事件
- 不合法则显示阻断原因

### 5. 状态日志区

记录最近 10~20 条事件：

- 时间
- 事件名
- 执行结果
- 当前状态
- 备注

日志用于回溯流程是否按预期运行。

## 状态模型

建议采用统一的状态机模型，字段至少包含：

- `status`：当前状态标识
- `orderNo`：分配号 / 订单号
- `storeId`：门店 ID
- `items`：当前菜品列表
- `totalAmount`：当前金额
- `callStaff`：是否已呼叫店员
- `paymentQrVisible`：是否展示支付二维码
- `paymentStatus`：支付状态
- `history`：事件历史
- `validationMessage`：最近一次校验提示

状态枚举建议：

- `ENTERED`
- `SELECTING`
- `BOUND`
- `COOKING`
- `MINIGAME`
- `ADD_DISH_PENDING`
- `REMOVE_DISH_PENDING`
- `CALL_STAFF`
- `STAFF_ADD_REMOVE`
- `STAFF_SETTLE`
- `STAFF_CONFIRM_AMOUNT`
- `QR_READY`
- `PAYING`
- `PAID`
- `FAILED`

## 校验规则

### 进入条件

- 进店后才能选菜
- 选菜后才能绑定或确认分配号
- 绑定成功后才能进入制作中
- 制作中后才能进入小游戏互动
- 小游戏互动中允许加菜 / 退菜 / 呼叫店员
- 呼叫店员后进入店员处理态
- 店员确认金额后才允许生成二维码
- 二维码生成后才允许支付
- 支付成功后进入完成态

### 动作合法性

每个动作都必须校验当前状态：

- 未绑定时不能直接进入制作中
- 未选菜时不能发起加菜/退菜
- 未呼叫店员时不能进行店员端加菜/退菜
- 未确认金额时不能出码
- 金额未确认时不能支付
- 支付完成后禁止再次修改订单

### 价格与数量校验

- 加菜后订单金额必须增加
- 退菜后订单金额必须减少或回滚
- 店员确认金额前，顾客端显示的金额与店员端确认金额保持一致
- 若金额不一致，显示提示并阻止出码

### 支付校验

- 若支付成功，订单状态改为 `PAID`
- 若支付失败，允许重试但保留订单信息
- 支付后不可再次加菜或退菜

## 场景扩展建议

本期在结构上预留以下场景入口，但不强制一次实现到最复杂：

1. 未绑定 / 错号 / 重复绑定
2. 加菜后重新算账
3. 退菜失败或库存不足
4. 店员拒绝修改订单
5. 金额不一致提示
6. 支付失败重试
7. 超时未操作提醒
8. 多人共用一单 / 桌台场景
9. 催菜 / 取餐提醒
10. 活动券 / 优惠券抵扣
11. 离店后评价入口

这些功能统一作为“可扩展场景卡片”展示，以免主流程过于拥挤。

## 页面布局建议

采用三段式布局：

1. 顶部：门店与订单状态摘要
2. 中部：步骤条 + 状态详情双栏布局
3. 底部：事件模拟器 + 状态日志

移动端可改为：

- 步骤条折叠
- 状态详情卡片置顶
- 事件按钮横向滚动
- 日志折叠展示

## 验收标准

至少满足以下条件即认为设计达标：

1. 能从“进店”一路模拟到“支付成功”。
2. 任一非法动作都会被阻止并显示原因。
3. 加菜、退菜后金额会正确变化。
4. 店员确认金额后才能出二维码。
5. 支付成功后状态锁定，禁止再次修改。
6. 日志可回溯每一次状态变化。
7. 页面可继续扩展更多业务场景而不破坏主流程。

## 实现约束

- 优先复用当前首页页面，不额外拆分复杂页面结构。
- 状态机逻辑应尽量独立于 UI，方便后续测试。
- 业务动作与展示动作分离，避免点击 UI 直接修改核心数据。
- 不引入真实后端依赖，默认使用本地状态机和可选 mock 数据。

## 推荐方案

采用“步骤面板 + 事件模拟器 + 状态日志”的单页方案。

原因：

- 最适合检验业务流程
- 交互清晰，易于演示和验收
- 对后续增加异常分支和场景扩展最友好
- 代码结构可独立于现有首页概览继续演进

## 后续实施拆分建议

实施时建议拆成三层：

1. 状态模型层：维护流程状态、校验规则、金额计算、事件历史
2. 页面交互层：步骤条、按钮、详情区、日志区
3. 场景扩展层：异常分支、活动券、多人桌台、提醒类功能
